Test Parser

Name

Test Parser — Visualizes a test framework's results.

Description

I built Test Parser to help me visualize test results. I designed it to be extensible so that I or someone else could later add new test frameworks and/or user interfaces. This way, whenever you end up working with a new framework that does not provide a nice-clean interface, you only have to write a parser to utilize what has been already written previously. Test Parser is intended for a primarily developer audience, but I can see it being used in other situations.

Note

When running, Test Parser will output the runner's stderr output to the console (stderr), so you can see what your runner is printing out. Test Parser will also output some stuff to stderr, but I currently have most of it logging to TestParser.log.

History

I built Test Parser to help me visualize/read the test results from Boost's Test framework. Before I realized that it could easily be extended to other frameworks, I called it Boost Test Parser. But since there is no affiliation with Boost, I've since changed the name.

Options

--help

Display help information.

--ui <ui framework>

Use specified ui.

  • qt

  • tkinter

  • text (default)

--view <view>

Use the specified views. Multiple vies can be specified in the following manner: 

--view=V1 --view=V1 --view=V2

Views earlier in the command have a higher priority. This means they are created earlier, thus will be updated earlier as well.

  • result (default)

  • statistic

--framework <test framework>

Use specified framework. See below for allowed frameworks.

--autoexpand on/off

Enable/disable auto-expansion

--threading

Enable threading. Note that threading is superfluous in most situations. It also leads to some bugs.

Version Info

Note that the Test Parser versions are when particular framework versions were introduced. So it is expected that those framework versions will work in later versions.

I have no plans to support other test formats, unless asked or I find the need. So if you desire a particular format, either notify me with the relevant information or feel free to implement yourself (fork the github repository.) If you plan on implementing yourself, you might want to check github issues to see if I have any plans on implementing that framework already or if I plan on making any changes that will complicate matters / break your code.

Python Unittest

Because of how Python Unittest is designed, we require that the test runner be setup a certain way. For parsing to work, we need the verbosity level to be set to 2. For running to work, we need the information all output to stdout. I.e: use the following command in your test runner. 

unittest.TextTestRunner(verbosity=2, stream=sys.stdout).run(suite)

JUnit

The JUnit framework is currently just JUnit output, not JUnit through Ant. Because this output doesn't contain more information about the passing tests, you will see 'free-floating' test cases in the output will little information. Also, the user needs to setup CLASSPATH and pass in the full java command (in quotes): 

java junit.textui.TestRunner className

for JUnit3 and 

java org.junit.runner.JUnitCore className

for JUnit4. Its possible to run a JUnit3 program with JUnit4's command, or at least I've seen it work. TestParser doesn't care which you use, as long as it works. If it doesn't work you will likely see JUnit's error message appear in stderr.

Auto Expansion

Without auto expand, when you rerun the tests the display tree will collapse. You will then have to manually go through and expand each node to reach your desired node. And if you rerun multiple times, you'll have to go through this process multiple times.

Auto expand seeks to remedy this problem. There are two ways to use auto expand. First is to select the nodes you want to be expanded/displayed after rerunning all the tests. Second is to simply select no tests and all the nodes that are currently displayed will be redisplayed after rerunning.

Qt Auto-expand + Threading

When used with threading enabled, qt autoexpand has a couple bugs. The biggest bug is that the program will sometimes seg-fault and crash. The other bug is that the root node, and sometimes other nodes, will not expand. Because of these bugs and the fact that threading hasn't been particularly useful, I suggest leaving threading disabled.

After auto-expanding, the scroll position will be reset to the position prior to rerunning.

Tk auto-scroll

Sometimes TkResultView's scroll position won't be set to the correct position. This is especially noticable when the tree is scrolled to the very end. As it turns out, this bug doesn't have anything to do with the auto-scroll, as it is still present even when auto-scroll is disabled.

Installation

Test Parser is packaged via distutils. To install, run setup.py install. I'm currently only using source distributions, but I'll try to add installers/packages later (maybe.)

Test Parser's installation is defaulted to install all documentation to /usr/share/doc/Test_Parser. Because of the directory location, you'll need to run the install as root. Or you can change the directory to somewhere else. To change the directory, either edit setup.cfg or pass in --install-data = <new_directory>. Windows users will have to change the directory because the default is designed for *nix systems.

Requirements

  • python 3.x

  • PyQt4 (Qt Views)

  • Tkinter (TK Views)

  • PLY 3.3 (JUnit)

If you do not wish to use the features that depend upon a library, you do not need to install it.

PyQt4 installation

Because there wasn't a PyQt package available for python3 when I was developing I had to compile and install PyQt4 manually. If when you're trying to install Test Parser check to see if there is a package available. It'll be something like: python3-Qt4.

I followed the instructions found here. To quote:

 

Download sip and pyqt from ... the links on the website: http://www.riverbankcomputing.co.uk/news

Extract the .tar.gz files to, say, your home folder, yielding the following folders:

PyQt-x11-gpl-4.6.2
sip-4.9.3

Follow the install directions. That is, in each folder, run: 

python3.1 configure.py
make
sudo make install
 
 --Edward

Tkinter installation

Tkinter is usually packaged with the python distribution. *nix users will probably want to get it from their favorite repository. Windows users will probably install TKinter during their python installation, but I don't know for sure.

PLY installation

PLY (Python Lex Yacc) is a python package that can be installed in the typical ways. Just download and run setup.py install. Make sure to install it with the correct python version, the correct version being the one used by TestParser (see above.)
