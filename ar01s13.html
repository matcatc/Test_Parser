Technical Details

On a high level, Test Parser simply runs the tests, parses them, and displays the results. There is more to it than this, mostly because I structured the code so that it is modular and can easily be extended. Of course this complicates the details, but the overall idea is the same.

Note

This technical documentation is likely outdated and incomplete. If it doesn't answer something you would like to know or you think should be added, please let me know (<matcatprg@yahoo.com>)

Test Results

Test Results is now structured using the Composite Pattern. Parsing code hasn't changed, and I don't think the parsing code will ever deal with the more abstract/generic info. But the UI code is now more generalized, thus allowing it to handle changes that may occur in Test Results in the future without having to change. UI code just takes an abstract object, and asks for its data and children. It then parses/ displays the data and recurses on the children.

The net result is that UI code does not make any assumptions about the type of object and thus more Test Result classes can be added later without breaking UI code.

See TestResults.dia for more information.

Documentation

There are two main parts to the documentation:

  • doxygen

  • docbook

The doxygen output is built from the documentation inlined in the code. Its more for developers than end users. I setup doxygen to use doxypy according to instructions I found here and here.

The docbook is more for end-users, but still contains information for developers. This file is built from docbook.

A script was provided in order to build the documentation and is aptly named: build_documentation.sh. While this script can be invoked directly, it is also used by setup.py. Setup.py has been customized so that when sdist is invoked, it will call build_documentation.sh and thus build all the documentation prior to packaging. As of June 10, 2010, all of the documentation is entirely rebuilt on each sdist invocation. Although I plan on designing it (probably through make or similar) so that it will only rebuild necessary files. At least if it gets to the point where it wastes too much time.

Note

build_documentation.sh needs to be run from the directory in which it is located.

Also included in the doc directory are UML diagrams and other miscellaneous and likely outdated documents.

Tests

The tests used to test Test Parser are packaged along with it. This way anyone, including users, can run them on their machine. To run all of the tests, run test_runner.py.

Possible reasons for a test to fail even though the code is fine:

  • Data files

    One of the data files used in testing (e.g: Boost_Test, xml) is missing or cannot be accessed. If its missing, its probably b/c we accidentally forgot to package it up with everything else. The latter, can happen and isn't really worth fixing. For instance, Boost_Test cannot be accessed when the application has been installed in a virtualenv. I haven't tried yet, but its also possible/likely that the same is true for a normal installation.
